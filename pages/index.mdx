## 왜 만들었나요?

프론트엔드를 혼자 담당하다 보니, 디자이너·백엔드와 소통하는 데 생각보다 많은 시간이 들었어요. <br />UI가 바뀔 때마다 직접 확인을 요청하는 대신, <br /> Storybook으로 컴포넌트를 문서화해서 props 옵션과 사용 예시를 바로 볼 수 있게 했거든요. <br /> <br />
그런데 Storybook만으로는 뭔가 부족했어요. <br /> "이 컴포넌트가 어떻게 생겼는지"는 보여줄 수 있어도, "왜 이렇게 설계했는지"는 담기 어려웠거든요. <br /> <br />
그래서 한 걸음 더 나아가기로 했어요. 어떤 문제를 해결하려고 이 컴포넌트를 만들었는지, <br />어떤 상황을 상정하고 props를 설계했는지를 글로 정리해두면 <br />나중에 나 자신에게도, 함께 일하는 사람에게도 훨씬 명확한 근거가 생긴다고 생각했어요.

## Inbom DS가 지향하는 것

1. **일관성** — 어떤 페이지에서든 같은 컴포넌트는 같은 모습으로 동작해요.

2. **재사용성** — 한 번 만든 컴포넌트는 어디서든 가져다 쓸 수 있어요.

3. **확장성** — props 하나로 다양한 상황에 대응할 수 있도록 설계했어요.

## 구성

### Foundation

디자인 시스템의 뼈대가 되는 토큰을 정의했어요. <br />색상, 타이포그래피, 간격, 그림자 모두 CSS 변수로 관리해서, 값 하나를 바꾸면 전체에 반영돼요.

### Components

인봄 플랫폼에서 실제로 사용한 컴포넌트들이에요. <br /> 각 컴포넌트마다 왜 이렇게 설계했는지, 어떤 상황에서 썼는지를 함께 담았어요.

| 카테고리 | 컴포넌트                                                          |
| -------- | ----------------------------------------------------------------- |
| Button   | Button, SocialButton, Switch, SocialBar                           |
| Badge    | Badge, BadgeGroup                                                 |
| Input    | Input, PasswordInput, SearchInput, Select, DatePicker, TimePicker |
| Modal    | Modal, ConfirmModal, Notification, BottomSheet                    |

## 기술 스택

### 인봄 플랫폼

- React — 컴포넌트 구현
- SCSS Modules — 컴포넌트 스타일링
- Storybook — 컴포넌트 문서화 및 디자이너 협업

### Inbom DS 문서 사이트

- Next.js + Nextra — 디자인 시스템 문서 사이트
- TypeScript — 타입 안정성
- SCSS Modules — 스타일링
